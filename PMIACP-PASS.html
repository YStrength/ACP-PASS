Part 1

敏捷心法与哲学：ACP 的灵魂

ACP 考试大量情景题其实就是在问：在这个情境下，最敏捷、最符合宣言和原则 的做法是什么？ 所以必须先把心法吃透。

核心 · Agile Manifesto

四大价值观（务必内化在脑子里）

  1. 个体和互动 高于 流程和工具
  2. 可工作的软件 高于 详尽的文档
  3. 客户合作 高于 合同谈判
  4. 响应变化 高于 遵循计划
考试技巧：冲突时优先选左边的值 但右边不是不要，而是“次之”
解题提醒： 如果选项体现的是“死守合同、流程、计划、文档”， 而忽视了人、沟通和价值交付，大概率就是错误选项。
案例 · Why

为什么“个体和互动”优先于流程和工具？

某团队遇到 UI 设计与前端实现严重不一致的问题。项目经理一开始要求大家严格按照 需求文档和流程走，但问题反复出现；敏捷教练介入后，立刻让 UI 与前端 面对面坐在一起画原型、看代码，半天内把问题解决。

这说明：人和沟通的效率 > 再精细的流程。流程和工具是辅助，而不是主角。

  • 复杂问题 → 依赖高质量沟通，而不是模板。
  • 多方理解偏差 → 互动才是校准认知的关键。
  • 考试中遇到“加强文档和流程” vs “让相关角色直接协作沟通”，优先选后者。
Principles

十二原则的“压缩版记忆”

  • 价值优先： 以满足客户为最高目标，持续交付有价值增量。
  • 拥抱变化： 欢迎需求变更，哪怕在后期。
  • 频繁交付： 以周为单位迭代，越短反馈越快。
  • 业务+技术协作： 业务方每天都要和团队紧密合作。
  • 激励与信任： 给到支持、信任团队，而不是微观管理。
  • 面对面沟通： 是最有效的沟通方式。
  • 可工作软件： 是衡量进度的第一标准。
  • 可持续节奏： 长期稳定的步伐好过短期爆冲。
  • 技术卓越： 持续优秀的工程实践是敏捷的地基。
  • 简单： 尽量减少不必要的工作。
  • 自组织团队： 最好的架构、需求和设计出自自组织团队。
  • 持续反思： 定期回顾并改进自己的方式。
Lean & Kanban

精益与看板：消除浪费，让工作流动起来

精益七大浪费（记住几个典型的即可）：

  • 做了没人用的功能 → 过度生产。
  • 写一半的代码长期未测试 → 半成品库存。
  • 每次小改动都走复杂审批 → 过度流程。
  • 一个人同时做 3 个项目 → 任务切换浪费。

看板关键三件事：

  • 可视化：任务放到看板上（如：待办/进行中/完成）。
  • 限制在制品（WIP）：每列最多几个任务。
  • 管理流动：观察哪里堵，就去疏通哪里。
高速公路收费站类比： 如果每个收费口都放车，出口却只有两车道 → 就堵死； 只打开 2 个收费口（WIP=2），整体通行反而更快。
Part 2

角色与团队：敏捷项目的人和组织

ACP 非常重视你对 角色职责、团队特征、仆人式领导 的理解。 操作层面的工具可以背，但对“人”的理解必须真正吃透。

Role · Product Owner

产品负责人（PO）：做“什么”的决策者

  • 定义产品愿景 → 明确“我们要去哪儿”。
  • 维护 Product Backlog，负责内容+优先级。
  • 对“价值最大化”负责，而非对“人天利用率”负责。
  • 和业务/客户持续沟通，是团队的“业务代表”。

考试倾向： PO 是 backlog 的唯一 owner，别人可以提供意见，不能跳过 PO 私自改优先级。

Role · Scrum Master

Scrum Master / 敏捷教练：团队的“守护神”

  • 推广敏捷价值观，辅导团队理解 Scrum / Kanban 等。
  • 移除障碍（跨部门协作、环境问题、流程瓶颈）。
  • 引导会议：站会、评审、回顾，而不是“主持批斗会”。
  • 典型形象是 仆人式领导（Servant Leader），而非“命令官”。
情景题判断： 项目经理/SM 的正确行为通常是 “帮助团队解决问题、保护团队免受干扰、促进沟通”， 而不是“强制分配任务、单方面制定计划、指责团队”。
Role · Team

开发团队：自组织、跨职能、T 型人才

  • 自组织（Self‑organizing）：自己规划如何完成迭代目标。
  • 跨职能（Cross‑functional）：团队内具备交付所需的各种技能。
  • T型人才：一专多能（有一个深度领域，有多个广度领域）。
  • 迭代承诺由团队给出，而不是别人拍脑袋指定。

考试常问： 当经理想“按人头分配任务”时，敏捷的回答是： 让团队自己决定如何拆分和分配工作。

Case · Servant Leadership

案例：仆人式领导 vs 传统项目经理

场景： 测试环境不稳定，团队无法进行集成测试。

传统式： “你们怎么还没搞定？写报告，周会上解释！” → 团队压力大、问题仍然存在。

仆人式： Scrum Master 说：“测试环境的问题交给我，你们专心做开发和自动化测试。” → 他去协调运维部门，争取了专门支持时间，并和团队一起在回顾会中思考如何避免类似问题（如增加预检脚本）。

启示： 仆人式领导不是“帮大家干活”，而是 为团队成功铺路。

Team Culture

安全环境：没有安全，就没有真正的回顾

迭代回顾会有一句著名的 Prime Directive（首要宣言）：

「无论我们发现了什么，都要相信大家在当时的认知和条件下，已经尽了最大努力。」

这句话的意义在于：让大家敢说真话，敢暴露问题，而不是把回顾会开成“甩锅大会”。这样团队才能找到真正的根因，并通过实验不断改进。

Part 3

工具与方法：ACP 高频必考“招式库”

本节包含 ACP 中最常被考到的工具和方法：用户故事、故事点、规划扑克、看板、WIP 限制、 迭代实践、回顾等，并配有小案例和“为什么”的解读。

User Story

用户故事：以“人”和“价值”为中心的需求

基本格式：

作为一个<角色>，我想要<某个功能>，以便<实现某个价值>。

示例： “作为一个网购用户，我想在商品页看到‘一键购买’按钮，以便可以快速完成下单。”

INVEST 原则：

  • Independent：互相独立，尽量少依赖。
  • Negotiable：细节可通过讨论确定，而非写死。
  • Valuable：对用户/业务有价值。
  • Estimable：可估算工作量。
  • Small：足够小，可以一个迭代完成。
  • Testable：有明确验收标准，可验证完成与否。
考试点： 遇到“需求文档写得很厚但没人看”时，敏捷的改进方向通常是 改用用户故事 + 持续对话 + 频繁演示。
Story Points

故事点：不要问“几小时”，要问“多大”

概念： 故事点是对工作项 相对规模/复杂度/不确定性 的估算单位，而不是时间。

常见序列： 1, 2, 3, 5, 8, 13, 20 ...（类斐波那契数列）。

小案例：

  • “登录”功能 → 2 点作为基准。
  • “注册”功能 → 比登录复杂很多 ≈ 8 点。
  • “重置密码” → 比登录稍复杂 ≈ 3 点。
Why： 时间估算容易受个人经验影响，而“相对大小”更容易达成共识， 也能通过 Velocity（速率） 在长期内转化为对进度的可靠预测。
Planning Poker

规划扑克：用游戏方式达成估算共识

  • PO 讲解用户故事。
  • 每位成员独立选一张带点数的牌（如 3 / 5 / 8）。
  • 一起亮牌，如点数差异大，则讨论“为什么你觉得小/为什么你觉得大”。
  • 讨论后重新亮牌，直到收敛。

收益： 暴露理解偏差、借助集体智慧校准估算。

Kanban & WIP

看板 & WIP 限制：让“完成”比“开始”更重要

看板列举例： 待办 → 进行中 → 测试 → 完成。

WIP 限制例： “进行中”列最多 3 个任务，“测试”列最多 2 个任务。

核心思想： 当某列达到 WIP 上限时，团队不再拉新任务，而是优先帮助这列任务向后流动。

考试典型情景： 测试排队、Bug 堆积、周期很长 → 敏捷做法是 限制在制品 + 优先清理瓶颈，而不是简单增加压力或加班。

Daily & Review

每日站会、演示会与回顾会

每日站会（Daily Stand-up）：

  • 三个问题：昨天做了什么？今天准备做什么？遇到什么障碍？
  • 目标是 同步信息 + 暴露问题，不是汇报工时。

迭代演示会（Review / Demo）： 把这次迭代完成的功能 Demo 给客户/PO 看，收集反馈，并更新 Backlog。

迭代回顾会（Retrospective）： 团队内部反省“哪些做得好/哪些可以改进/下次要尝试什么新做法”，是持续改进引擎。

考试要点： 遇到“项目一团乱，问题重复出现，团队士气低”，正确动作通常是 加强回顾、建立安全氛围、尝试小实验改进，而不是“增加检查表、加强惩罚”。

Adaptive Planning

适应性规划：滚动式而非一次性大爆炸

  • 长期只做粗略规划 → 保持灵活。
  • 中期（下几次迭代）做较详细规划。
  • 最近一次迭代做最详细规划（任务级）。
  • 每个迭代结束后，根据新信息 重新规划。

考试场景： 需求高度不确定、变化频繁 → 采用迭代+增量+滚动规划，而不是一次性锁定全部范围。

Part 4

度量与图示：燃尽图、燃起图、看板、累积流图

ACP 不仅考概念，也会考你对常见敏捷图表的理解。这里用 示意图 + 解读 的方式帮你记住它们， 并能在考场上快速识别。

Chart · Burndown

燃尽图（Burndown Chart）示例

横轴： 时间（迭代天数）。
纵轴： 剩余工作量（故事点）。

示例：团队起始剩余 40 点，实际燃尽线在前期略慢，后期加速趋近理想线。

解题关键： 实际曲线长期高于理想线 → 有进度风险； 曲线“平”很久 → 说明很久没完成任何故事，可能存在隐形瓶颈。

Chart · Burnup

燃起图（Burnup Chart）示例

横轴： 时间。
纵轴： 已完成工作量（故事点）。

燃起图更适合观察 范围变化：如果总范围线往上跳，说明中途加了需求。

考试点： 涉及“范围频繁变化”时，燃起图比燃尽图更清晰；总范围线不断上升 → 范围蔓延风险。

Chart · Kanban

简单看板示意图

典型列：待办 → 进行中 → 测试 → 完成，每列有 WIP 限制。

在考试题中，如果“进行中/测试”列任务爆棚，敏捷建议是 引入/收紧 WIP 限制 + 团队一起清理瓶颈。
Chart · CFD

累积流量图（CFD）示例

CFD 展示不同状态的工作项数量随时间的变化，用来识别瓶颈和交付节奏。

“进行中”区域突然变宽 → WIP 上升，说明该阶段成了瓶颈；
“已完成”区域斜率越大 → 交付速度越快。
Part 5

核心术语速查：ACP 最常考的概念解释

本节整理了 ACP 考试中最典型、最容易混淆的术语和概念。建议在刷题时常回来查阅这一节。

Glossary

高频术语解释（精炼版）

  • 用户故事（User Story）
    以“作为一个…我想…以便…”格式描述的需求，强调用户价值和对话，而不是大篇技术细节。
  • 故事点（Story Point）
    相对估算单位，综合考虑复杂度、工作量和不确定性，常用 1/2/3/5/8/13…序列。不是小时。
  • 速率（Velocity）
    每个迭代团队实际完成的故事点总数，用于预测未来迭代可完成工作量，但不用于比较团队绩效。
  • 在制品限制（WIP Limit）
    限制每个流程状态下同时进行的任务数，用于暴露瓶颈、减少多任务切换、加快整体流动。
  • 燃尽图 / 燃起图（Burndown / Burnup）
    燃尽图关注“剩余工作量”，燃起图关注“已完成工作量”和“范围变化”；范围经常变化时用燃起图更清晰。
  • 累积流量图（CFD）
    展示各状态任务数量随时间变化的图，常用于识别流程瓶颈和估算交付周期。
  • 仆人式领导（Servant Leadership）
    领导通过服务团队来实现更高绩效，关注移除障碍、赋能和培养团队，而不是命令和控制。
  • 自组织团队（Self‑organizing Team）
    团队根据共同目标，自行决定如何分工和协作，外部不进行微观管理。
  • 跨职能团队（Cross‑functional Team）
    团队具备交付工作所需的各种技能（需求、设计、开发、测试…），减少跨部门依赖。
  • T 型人才 / 通才型专家（T‑shaped）
    在某一领域有深度（T 的竖），在多个相关领域有广度（T 的横），适合敏捷团队的协作模式。
  • 迭代 / 冲刺（Iteration / Sprint）
    固定时间盒（通常 1–4 周），在此期间团队完成一组可交付的增量成果。
  • 产品待办列表（Product Backlog）
    PO 维护的需求总清单，包含所有待开发的功能项，并按价值/风险/优先级排序。
  • 迭代待办列表（Sprint Backlog）
    当前迭代中要完成的工作项，是团队根据速率和目标主动承诺的计划。
  • ATDD / TDD / BDD
    ATDD： 从验收测试出发驱动开发；
    TDD： 先写单元测试再写实现代码；
    BDD： 用业务可读语言描述行为驱动开发。
  • 最小可行产品（MVP）
    最小功能集，却能验证核心假设、开始创造价值的产品版本，用来快速获取反馈。
Part 6

ACP 考试攻略：如何把这些知识转化为 4A 成绩

知道“是什么”和“为什么”之后，还需要知道：在考试时如何选对答案。本节给你一套可执行的策略。

Strategy · Mindset

1. 先用“敏捷过滤器”看选项

看到一道情景题，先不要急着看四个选项，先问自己：

  • 这个情景下，敏捷会优先考虑什么？（人、价值、反馈、协作、适应性）
  • 哪种做法最符合 四大价值观和十二原则？

然后再看选项，把明显不敏捷的做法删掉，例如：

  • “要求团队严格按原计划执行，不允许改变范围。”
  • “增加更多审批和文档来控制团队。”
  • “领导单方面做决定，不需要团队参与。”
记忆口诀： 感觉像“官僚主义、层层审批、追责文化”的选项，大多不是敏捷答案。
Strategy · People

2. 关于人的题：优先仆人式领导 + 协作 + 安全

当题目涉及“项目经理 / SM 下一步应该怎么做？”，优先考虑：

  • 帮助团队解决障碍（去协调整个系统，而不是让团队硬扛）。
  • 促进干系人之间的沟通和协作（拉人坐在一起，而不是隔着文档吵）。
  • 建立安全、信任的环境（鼓励暴露问题，而不是惩罚失误）。

反之，如果选项体现的是：

  • 责怪个体、公开批评、威胁惩罚。
  • 越级决策、忽视团队意见。
  • 强调个人绩效而非团队绩效。

那么该选项通常不是敏捷推荐的答案。

Strategy · Tools

3. 工具相关题：记住每种图表解决什么问题

  • 燃尽图 → 看“剩余量是否按计划下降”，关注进度风险。
  • 燃起图 → 看“完成量 + 范围变化”，适用于需求变化多的场景。
  • 看板 + WIP → 关注“工作流是否顺畅”，用于处理瓶颈和多任务。
  • CFD → 识别“哪个状态的工作堆积”，用于发现流程问题。

凡是提到“提高流动效率、减少多任务、暴露瓶颈”，就可以想到 WIP 和看板。
